# 产品商用可行性审计技能 (Product Production-Readiness Audit)

触发此技能后，你将以**拥有 10 年经验的技术架构师兼安全工程师**的视角，对指定代码模块进行全链路的商用可行性审计。

---

## 审计流程

### 第一步：扫描结构

1. 使用 `list_dir` 递归扫描目标目录，建立完整的文件树
2. 识别关键文件类型：入口文件（app.py）、配置（settings/config）、核心业务逻辑、数据层、工具层

### 第二步：深度读取核心文件

按以下优先级逐一读取：
1. **配置文件**（settings.py / config.py）：检查环境变量、硬编码、路径安全
2. **入口文件**（app.py）：检查异常处理、鉴权、限流
3. **核心逻辑**（业务处理层）：检查并发安全、内存泄漏、数据流完整性
4. **数据层**（ORM / 文件 IO）：检查事务、锁机制、文件操作安全

### 第三步：输出结构化报告

报告必须严格按照以下结构输出，**并将文档保存至** `document/03-产品审计报告/` 目录：

```
document/03-产品审计报告/
└── YYYYMMDD_HHMMSS_[模块名]_商用可行性审计.md
```

## 审计重点检查清单

### 安全性
- [ ] 是否存在路径穿越漏洞（../）
- [ ] 用户输入是否有白名单校验
- [ ] 敏感信息是否通过环境变量注入（不硬编码）
- [ ] 是否有未经授权的接口访问风险

### 稳定性
- [ ] 全局异常是否有兜底捕获
- [ ] 是否存在内存泄漏风险（如全局字典无限增长）
- [ ] 并发场景下是否存在竞争条件（线程锁）
- [ ] 第三方依赖是否版本锁定

### 可扩展性
- [ ] 配置是否统一收口（避免硬编码分散）
- [ ] 是否具备水平扩展能力
- [ ] 向量库/数据库连接是否单例化

### 成本控制
- [ ] 是否存在不必要的重复 LLM 调用
- [ ] 是否有限速/限流机制
- [ ] API Key 是否有用量告警机制

### 运维可观测性
- [ ] 关键操作是否有结构化日志
- [ ] 是否有推理延迟监控
- [ ] 错误是否有告警通知机制

---

## 评分标准

| 评级 | 标准 |
| :--- | :--- |
| ✅ 可商用 | 无高危问题，中危问题 ≤ 2 个 |
| ⚠️ 需加固 | 存在 1-3 个高危问题，或中危问题 ≥ 3 个 |
| 🔴 不可商用 | 存在 4 个以上高危问题，或存在数据安全漏洞 |

---

## 特别提示

- 審計完成后，**主动询问**用户是否需要直接帮助修复高危问题
- 修复时遵循项目规则：全程使用简体中文注释，强制使用 pathlib 处理路径
- 每次修复后在路线图中更新状态为 `[已修复 YYYYMMDD]`
- **闭环归档**：在所有高危问题修复并验证完毕后，**必须主动提示用户**使用 `@qa_summarizer` 或 `@qa_archiver` 对本次加固过程进行技术沉淀归档。
