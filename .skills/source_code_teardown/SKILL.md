# 源码底层拆解分析规范 (Source Code Teardown Guidelines)

当用户引入此 Skill 并要求进行代码问题的“源码拆解”、“源码分析”、“原理解析”或需要你深入了解某个底层机制时，请严格按照以下结构、深度和格式进行内容输出，为用户提供真正的“上帝视角”。

## 1. 核心原则
不再只是简单地粘贴源码、做表面翻译或逐行解释，而是**必须深挖底层运行机制**，梳理组件间错综复杂的依赖关系、状态扭转和信息流转机制，并最终推导出对实际开发有明确指导意义的最佳工程实践。

## 2. 强制输出的大纲结构

每一次全面的源码拆解分析，**必须**完整包含以下六个核心模块：

### 模块一：全局架构设计 (Global Architecture Design)
- **内容要求**：从宏观视角说明核心类、接口与组件之间的静态依赖关系。
- **格式要求**：**必须**包含一段 Mermaid `classDiagram` (类图)。
- **分析重点**：简明扼要地解释各个核心组件在整个架构中扮演的抽象角色（例如：谁是代理拦截器、谁是工厂钩子、谁负责物理存储、谁作为数据载体）。

### 模块二：宏观时序流转图 (Macro Sequence Flow)
- **内容要求**：展示一次典型调用时，核心参数和上下文信息在各个相关组件间是如何一来一回流转配合的。**必须体现用户调用、中间拦截器、核心执行逻辑与存储介质之间的交互路径。**
- **格式要求**：**必须**包含一段 Mermaid `sequenceDiagram` (时序图)。
- **分析重点**：理清“调用预处理 -> 核心逻辑执行 -> I/O 后处理”的严谨时序边界与触发点。

### 模块三：生命周期逻辑流程图 (Lifecycle & Logic Flow)
- **内容要求**：聚焦于核心触发从属逻辑、执行环境准备、数据读取、推理及落盘的完整逻辑链条。
- **格式要求**：**必须**包含一段 Mermaid `graph TD` (流程图)。通过 subgraph 划分出明显的生命周期阶段（例如：环境准备阶段、读取/转换阶段、持久化阶段）。
- **分析重点**：展示异常分支、钩子触发时机以及执行状态如何在不同阶段切换。

### 模块四：核心机制源码深度拆解 (Core Code Deep Dive)
- **内容要求**：直接切入底层最关键的几段真实源码进行拆解。
- **格式要求**：
  - 代码块需要精简，剔除无关的错误处理代码，保留核心逻辑主干。
  - **必须**在源码中关键行上方使用 `# 1. ...`, `# 2. ...` 等数字标号进行原代码级别的注释。
  - 在代码块下方，针对这些数字标号进行详尽的“逐点深度剖析”。
- **分析重点**：必须点出原作者在此处的设计巧思（如利用字典进行隐式传参、闭包、单例锁等）或潜在的代码陷阱。

### 模块五：底层表现对比与性能优化 (Performance & Anti-patterns)
- **内容要求**：基于对底层源码的透彻理解，分析常见的使用误区与正确用法的性能和安全差异。
- **格式要求**：**强烈建议**使用 Markdown 表格进行清晰的双向对比（❌ 错误/常规方案 vs ✅ 极致/官方推荐方案）。
- **分析重点**：从物理底层表现（如网络请求次数、文件描述符打开次数、I/O 阻塞时长、内存占用峰值、O(N) 复杂度等）维度去揭示“为什么这么做会更好”。

### 模块六：工程卓越性总结与铁律 (Engineering Excellence Rules)
- **内容要求**：知其然更要知其所以然。将前面的分析提炼为我们在实际业务开发中必须遵守的原则。
- **格式要求**：以列表形式列出 3条左右 的工程规范“铁律”。
- **分析重点**：每一条铁律必须是可以落地执行的编码规范，直接指导之后的代码编写。

## 3. 沟通与输出基调
- **专业且透彻**：不要停留在“这个函数是用来做什么的”这种业务语境中，要使用“多态”、“实例挂载”、“序列化/反序列化”、“阻塞”、“代理模式”、“闭包捕获”等更接近框架底层的专业术语进行解释。
- **图文并茂**：不要通篇大段文字，保证 Mermaid 图表的视觉清晰度。
- **聚焦核心干货**：不要被细枝末节的非核心错误处理逻辑干扰，时刻把用户的注意力拉回“架构”和“信息流转”两个核心上。
