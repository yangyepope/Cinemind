# 问答总结提炼规范 (Q&A Summarizer Guidelines)

当用户引入此 Skill (`@qa_summarizer`) 并要求总结之前的问答对话时，请严格按照以下结构和规范进行内容提炼，确保产出的总结不是对话流水账的简单罗列，而是系统化、严谨清晰的知识沉淀。

## 1. 核心原则
- **滤除冗余，提炼干货**：去除对话中的寒暄、重复提问、中间试错和口语化表达，只保留核心痛点剖析与最终的知识结晶。
- **重构逻辑体系**：绝对不要按时间流水账形式（“你问了什么，我答了什么”）来排列。必须根据知识的关联度，重组为“现象/背景 -> 核心误区 -> 底层原理/破局切入 -> 结论与代码落地”的递进教学逻辑。
- **专业与结构化视觉**：使用精准的技术术语，强制运用 Markdown 多级标题、列表、引用块（Quote）、加粗强调以及格式化的高亮代码块，打造极致的阅读体验。

## 2. 强制输出结构大纲

每次进行问答记录的全盘总结时，必须将内容填充到以下五个模块中：

### 模块一：全局主题摘要 (Executive Summary)
- 简明扼要地概括本次长段对话的**核心探讨主题**及最终解决的**最大痛点/盲区**。
- 以标签形式列出本次问答的 3-5 个核心技术关键词。

### 模块二：认知盲区与核心问题重塑 (Key Questions & Misconceptions)
- 将用户原先零散、甚至表述不清的提问，重新归纳提炼为几个精准的“高价值核心问题”。
- 必须明确指出用户最初陷入的**错误认知**（例如：“原以为A是X，但其实A是Y”）。

### 模块三：底层原理解密 (Underlying Mechanisms & Principles)
- **底层原理深度讲透**：集中所有的笔墨，详细拆解解决这些问题的核心机制（要敢于深入源码、设计模式或编译层级的解释）。
- **对比与纠偏**：通过直观的正反对比（如“组装期 vs 运行期”），或生动的现实生活比喻（“建流水线车间 vs 拉电闸”），将艰深的原理通俗化。
- **关键源码佐证**：如果涉及代码原理，必须附带精简后的极简核心源码片段，并在代码内保留关键步骤的中文大白话注释。

### 模块四：最佳实践方案 (Best Practices & Solutions)
- 给出经过本轮探讨、最终验证跑通的、可以直接复制到工程里运行的**最佳代码范例**。
- 简述这段最终代码为什么比原始形态更好，好在哪里（高扩展、解耦、优雅等）。

### 模块五：工程心法与记忆铁律 (Key Takeaways)
- 将几千字的原理浓缩。
- 以无情、直接的口吻，列出 1-3 条开发者下次遇到同类问题应该“无脑”想起的技术心法或开发铁律。

## 3. 输出流转与落地要求
- **格式纪律**：输出内容必须强制使用标准 Markdown 语法的完整文章结构。
- **自动化持久储存**：生成总结后，**必须主动将该总结输出为一个独立的 Markdown 文件，保存至 `d:\1-application\python_project\Cinemind\document\01-LangChain源码底层解析` 目录下**。
- **自动排序命名**：在创建文件前，主动遍历目标文件夹内现有的数字前缀（如 `01-`、`02-`），为新总结文件分配下一个递增序号进行排号存盘（例如 `03-LCEL组件装配执行流转_QA复盘.md`）。
- **沟通语调**：语气需专业、系统化。保存文件完毕后，应在聊天框向用户汇报生成成功，并给出对应文件的系统绝对路径。
