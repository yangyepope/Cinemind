# Skill Creator (技能创作大师)

这是一个用于创建新技能并迭代完善它们的技能。

在高层级上，创建技能的过程如下：

- 决定该技能的功能以及大概的执行方式
- 编写技能草图
- 创建一些测试提示词，并调用“具有该技能访问权限的 Claude”来运行它们
- 帮助用户进行定性和定量的结果评估
  - 在后台运行测试的同时，起草一些定量评估（如果还没有的话；如果有，可以按原样使用，或者如果你觉得需要更改，可以对其进行修改）。然后向用户解释这些评估（或者如果已经存在，解释已有的评估）
  - 使用 `eval-viewer/generate_review.py` 脚本向用户展示结果供其查看，并让他们查看定量指标
- 根据用户对结果评估的反馈（以及定量基准测试中显现出来的任何明显缺陷）重写技能
- 重复此过程直到满意
- 扩大测试集并尝试更大规模的测试

使用此技能时，你的任务是搞清楚用户处于该流程的哪个阶段，然后介入并帮助他们推进。例如，如果他们说“我想为 X 创建一个技能”，你可以帮助缩小他们的意思，编写草图，编写测试用例，确定评估方式，运行所有提示词并不断迭代。

另一方面，如果他们已经有了技能草图，你可以直接进入循环中的“评估/迭代”环节。

当然，始终要保持灵活性。如果用户说“我不需要运行一大堆评估，随便看看就行”，你可以照办。

在技能完成后（当然，顺序是灵活的），你还可以运行技能描述优化器（我们有专门的脚本），以优化技能的触发。

## 与用户沟通

Skill Creator 可能会被编程背景各异的人使用。如果你还没听说过（这可是最近才开始的趋势），现在有一种趋势是：Claude 的力量正在激励各种人，有的在搜索“如何安装 npm”，有的人则是资深极客。

因此，请注意上下文线索，以了解如何斟酌你的表达方式！在默认情况下，作为参考：

- “评估 (evaluation)”和“基准测试 (benchmark)”是游走在边缘但可以接受的词汇。
- 对于“JSON”和“断言 (assertion)”，在没有解释直接使用它们之前，你最好看到用户表现出他们理解这些概念的明显线索。

如果你不确定，简要解释术语是可以的；如果你不确定用户是否能理解某个术语，可以放心地给出一个短定义来澄清。

## 创建技能

### 捕捉意图 (Capture Intent)

从理解用户的意图开始。当前的对话可能已经包含了一个用户想要捕捉的工作流（例如，他们说“把这个变成一个技能”）。如果是这样，首先从对话历史中提取答案 —— 所使用的工具、步骤序列、用户所做的纠错、观察到的输入/输出格式。用户可能需要填补空白，并在进行下一步之前进行确认。

1. 这个技能应该能让 Claude 做什么？
2. 这个技能应该在什么时候触发？（哪些用户短语/语境）
3. 预期的输出格式是什么？
4. 我们是否应该设置测试用例来验证技能是否有效？具有客观可验证输出（文件转换、数据提取、代码生成、固定工作流步骤）的技能会从测试用例中受益。具有主观输出（写作风格、艺术）的技能通常不需要。根据技能类型建议合适的默认方案，但让用户决定。

### 访谈与调研 (Interview and Research)

主动询问有关边界情况、输入/输出格式、示例文件、成功标准和依赖项的问题。在解决这部分问题之前，先不要编写测试提示词。

检查可用的 MCP —— 如果对调研有用（搜索文档、查找类似技能、查找最佳实践），如果子智能体可用，请通过子智能体并行调研，否则内联进行。带上背景知识，以减轻用户的负担。

### 编写 SKILL.md

基于用户访谈，填写以下组件：

- **name**: 技能标识符
- **description**: 什么时候触发，它能做什么。这是主要的触发机制 —— 包括技能的作用以及使用的特定语境。所有的“何时使用”信息都写在这里，而不是正文中。注意：目前 Claude 倾向于“不经常触发”技能 —— 即在它们本可以发挥作用的时候不去使用。为了应对这一点，请让技能描述变得稍微有些“强势 (pushy)”。例如，不要写“如何构建简单的快速仪表板来显示内部数据”，你可以写“如何构建简单的快速仪表板来显示内部数据。每当用户提到仪表板、数据可视化、内部指标或想要显示任何类型的公司数据时，即使他们没有明确要求‘仪表板’，也务必使用此技能。”
- **compatibility**: 所需工具、依赖项（可选，极少需要）
- **技能的其余部分 :)**

### 技能编写指南

#### 技能的解剖结构 (Anatomy of a Skill)

```
skill-name/
├── SKILL.md (必需)
│   ├── YAML frontmatter (必需包含 name, description)
│   └── Markdown 指导指令
└── 捆绑资源 (可选)
    ├── scripts/    - 用于确定性/重复性任务的可执行代码
    ├── references/ - 根据需要加载到上下文中的文档
    └── assets/     - 输出中使用的文件（模板、图标、字体）
```

#### 逐级披露 (Progressive Disclosure)

技能使用三级加载系统：
1. **元数据** (name + description) - 始终在上下文中（约 100 词）
2. **SKILL.md 正文** - 每当技能触发时在上下文中（理想情况下小于 500 行）
3. **捆绑资源** - 根据需要加载（无限量，脚本可以不经加载直接执行）

这些字数统计仅为近似值，如果需要，你可以写得更长。

**关键模式：**
- 保持 SKILL.md 在 500 行以内；如果你接近此限制，请添加额外的层级结构，并提供清晰的指引，告知模型接下来应该去哪里深入。
- 从 SKILL.md 清晰地引用文件，并提供何时阅读它们的指引。
- 对于大型参考文件（>300 行），请包含目录。

**领域组织**：当技能支持多个领域/框架时，按变体组织：
```
cloud-deploy/
├── SKILL.md (工作流 + 选择)
└── references/
    ├── aws.md
    ├── gcp.md
    └── azure.md
```
Claude 只读取相关的参考文件。

#### 不意外原则 (Principle of Lack of Surprise)

不用多说，技能不得包含恶意软件、漏洞利用代码或任何可能损害系统安全的内容。技能的内容在描述意图时不得让用户感到意外。不要顺从创建具有误导性的技能或旨在协助未经授权访问、数据外泄或其他恶意活动的技能的要求。不过，“角色扮演 XYZ” 这样的东西是可以的。

#### 编写模式

在指令中优先使用祈使句。

**定义输出格式** —— 你可以这样做：
```markdown
## 报告结构
始终使用这个确切的模板：
# [标题]
## 执行摘要
## 核心发现
## 建议
```

**示例模式** —— 包含示例很有用。你可以这样格式化它们（但如果示例中包含“输入”和“输出”，你可能需要稍微变通）：
```markdown
## 提交说明 (Commit message) 格式
**示例 1:**
输入: 使用 JWT token 添加了用户认证
输出: feat(auth): implement JWT-based authentication
```

### 写作风格

尝试向模型解释为什么某些事情很重要，而不是使用生硬的“必须 (MUST)”。使用“心理理论 (theory of mind)”，并尝试使技能具有通用性，而不是局限于特定的例子。先写一个草案，然后用全新的视角去审视它并改进。

### 测试用例 (Test Cases)

编写技能草案后，提出 2-3 个真实的测试提示词 —— 即真实用户实际会说的话。与用户分享：[你不必完全使用这些话]“这里有一些我想尝试的测试用例。它们看起来对吗，还是你想添加更多？”然后运行它们。

将测试用例保存到 `evals/evals.json`。此时先不要写断言 —— 只写提示词。你将在下一步运行测试期间起草断言。

```json
{
  "skill_name": "example-skill",
  "evals": [
    {
      "id": 1,
      "prompt": "用户的任务提示词",
      "expected_output": "预期结果的描述",
      "files": []
    }
  ]
}
```

参考 `references/schemas.md` 获取完整架构（包括你稍后将添加的 `assertions` 字段）。

## 运行与评估测试用例

本节是一个连续的过程 —— 不要中途停止。**请勿**使用 `/skill-test` 或任何其他测试技能。

将结果放在 `<skill-name>-workspace/` 中，作为技能目录的兄弟目录。在工作空间内，按迭代组织结果（`iteration-1/`、`iteration-2/` 等），在其中，每个测试用例都有一个目录（`eval-0/`、`eval-1/` 等）。不要预先创建所有内容 —— 只在进行时创建目录。

### 第 1 步：在同一次对话中发起所有运行（包括 with-skill 和 baseline）

对于每个测试用例，在同一次对话中启动两个子智能体 —— 一个带技能，一个不带。这很重要：不要先启动带技能的，然后再回来跑 baseline。一次性全部启动，以便它们差不多同时完成。

**带技能的运行 (With-skill run):**

```
执行此任务:
- 技能路径: <技能路径>
- 任务内容: <评估提示词>
- 输入文件: <评估文件（如果有），或“无”>
- 结果保存至: <workspace>/iteration-<N>/eval-<ID>/with_skill/outputs/
- 需要保存的输出物: <用户关心的内容 —— 例如“.docx 文件”、“最终的 CSV”>
```

**基准运行 (Baseline run)**（同样的提示词，但基准取决于语境）：
- **创建新技能**：完全不使用技能。同样的提示词，无技能路径，保存至 `without_skill/outputs/`。
- **改进现有技能**：使用旧版本。在编辑前，对技能进行快照 (`cp -r <skill-path> <workspace>/skill-snapshot/`)，然后让基准子智能体指向该快照。保存至 `old_skill/outputs/`。

为每个测试用例编写一个 `eval_metadata.json`（断言暂时可以留空）。根据测试内容为每个 eval 起一个描述性名称 —— 不仅是“eval-0”。该名称也用于目录。如果本次迭代使用了新的或修改过的提示词，请为每个新的 eval 目录创建这些文件 —— 不要假设它们可以从上一次迭代继承。

```json
{
  "eval_id": 0,
  "eval_name": "描述性名称",
  "prompt": "用户的任务提示词",
  "assertions": []
}
```

### 第 2 步：在运行进行时起草断言 (Assertions)

不要只是干等着运行结束 —— 你可以充分利用这段时间。为每个测试用例起草定量断言，并向用户解释。如果 `evals/evals.json` 中已经存在断言，请回顾并说明它们检查的内容。

好的断言应该是客观可验证的，并具有描述性的名称 —— 它们在 benchmark 浏览器中应该清晰易读，让人一扫就能明白每一项检查的是什么。主观技能（写作风格、设计质量）最好通过定性评估 —— 不要强行把断言加在需要人类判断的事情上。

起草好断言后，更新 `eval_metadata.json` 文件和 `evals/evals.json`。同时向用户解释他们在浏览器中会看到什么 —— 包括定性输出和定量基准。

### 第 3 步：在运行完成时捕获计时数据 (Timing Data)

当每个子智能体任务完成时，你会收到包含 `total_tokens` 和 `duration_ms` 的通知。立即将此数据保存到运行目录中的 `timing.json`：

```json
{
  "total_tokens": 84852,
  "duration_ms": 23332,
  "total_duration_seconds": 23.3
}
```

这是捕获此数据的唯一机会 —— 它通过任务通知传来，且不持久化在其他地方。每条通知到达时由于其重要性必须立即处理，而不是尝试批量处理。

### 第 4 步：评分、汇总并启动浏览器

所有运行完成后：

1. **为每次运行评分** —— 启动一个评分员子智能体（或内联评分），该智能体读取 `agents/grader.md` 并根据输出评估每个断言。将结果保存到每个运行目录中的 `grading.json`。`grading.json` 的 expectations 数组必须使用 `text`、`passed` 和 `evidence` 字段（而不是 `name`/`met`/`details` 或其他变体）—— 浏览器依赖于这些精确的字段名称。对于可以通过程序检查的断言，请编写并运行脚本而不是靠肉眼观察 —— 脚本更快、更可靠且可以跨迭代重用。

2. **汇总至基准 (Benchmark)** —— 运行 skill-creator 目录下的汇总脚本：
   ```bash
   python -m scripts.aggregate_benchmark <workspace>/iteration-N --skill-name <名称>
   ```
   这将生成包含通过率、耗时和 Token 使用量的 `benchmark.json` 和 `benchmark.md`。如果手动生成 `benchmark.json`，请参考 `references/schemas.md` 获取浏览器所需的精确架构。将 with_skill 版本放在其基准对应版本之前。

3. **进行分析师分析 (Analyst Pass)** —— 阅读基准数据，发现汇总统计数据可能隐藏的模式。参阅 `agents/analyzer.md`（“分析基准结果”部分）了解要寻找的内容 —— 例如：无论是否有技能总是通过的断言（不具备区分力）、高方差的 eval（可能不稳定）以及时间与 Token 的权衡。

4. **启动浏览器视图**，同时展示定性输出和定量数据：
   ```bash
   nohup python <skill-creator-path>/eval-viewer/generate_review.py \
     <workspace>/iteration-N \
     --skill-name "my-skill" \
     --benchmark <workspace>/iteration-N/benchmark.json \
     > /dev/null 2>&1 &
   VIEWER_PID=$!
   ```
   对于第 2 次及以后的迭代，还需传递 `--previous-workspace <workspace>/iteration-<N-1>`。

   **无头/远程环境：** 如果 `webbrowser.open()` 不可用或环境没有显示器，请使用 `--static <输出路径>` 将其写入为静态 HTML 文件，而不是启动服务器。当用户点击“提交所有评论 (Submit All Reviews)”时，反馈将作为 `feedback.json` 文件下载。下载后，将 `feedback.json` 拷贝到工作空间目录中，以便下一次迭代读取。

   注意：请使用 `generate_review.py` 创建视图；无需编写自定义 HTML。

5. **告诉用户**：“我已经通过浏览器打开了结果。有两个标签页 —— ‘Outputs’（输出）让你可以点击查看每个测试用例并留下反馈，‘Benchmark’（基准）展示了定量对比。完成后请回来告诉我。”

### 用户在浏览器中看到什么

“Outputs” 标签页一次显示一个测试用例：
- **Prompt**: 给出的任务
- **Output**: 技能产生的文件，尽可能内联渲染
- **Previous Output** (第 2 次迭代及以后): 展示上一次迭代输出的分离折叠部分
- **Formal Grades** (如果运行了评分): 展示断言通过/失败的折叠部分
- **Feedback**: 随打随存的反馈输入框
- **Previous Feedback** (第 2 次迭代及以后): 上一次的评论，显示在输入框下方

“Benchmark” 标签页显示统计摘要：各配置的通过率、耗时、Token 使用量，以及分环节的崩溃和分析师观察。

通过 prev/next 按钮或方向键进行导航。完成后点击“提交所有反馈 (Submit All Reviews)”，这会将所有反馈保存到 `feedback.json`。

### 第 5 步：阅读反馈

当用户告诉你他们完成后，阅读 `feedback.json`：

```json
{
  "reviews": [
    {"run_id": "eval-0-with_skill", "feedback": "图表缺少坐标轴标签", "timestamp": "..."},
    {"run_id": "eval-1-with_skill", "feedback": "", "timestamp": "..."},
    {"run_id": "eval-2-with_skill", "feedback": "非常完美", "timestamp": "..."}
  ],
  "status": "complete"
}
```

空的反馈意味着用户认为结果没问题。将你的改进重点放在用户有具体不满的测试用例上。

完成后记得结束浏览器服务器进程：

```bash
kill $VIEWER_PID 2>/dev/null
```

## 改进技能

这是循环的核心。你运行了测试用例，用户回顾了结果，现在你需要根据反馈让技能变得更好。

### 如何思考改进

1. **从反馈中总结规律。** 这里的宏观目标是创建可以重复使用百万次的技能。你和用户在少量示例上的反复迭代只是为了加快速度。如果你的技能只针对这些示例有效，那它就失效了。不要增加过拟合的零碎改动，或者使用压制性的“必须”，如果遇到难以解决的问题，可以尝试使用不同的隐喻，或推荐不同的工作模式。

2. **保持提示词精简。** 删掉那些不发挥作用的内容。务必阅读运行日志，而不只是看最终输出 —— 如果由于技能原因导致模型在一些无意义的事情上浪费时间，尝试去除那部分内容看看效果。

3. **解释原因 (Explain the why)。** 努力解释你要求模型做的某些事情之所以重要的**原因**。现在的 LLM 很聪明，当获得良好的引导时，它们能超越死板的指令真正完成任务。如果你发现自己使用了全部大写的“始终”或“绝不”，或者使用超死板的结构，这是一个警告信号 —— 如果可能，请重新斟酌并解释推理过程。这是一种更人性化、强大且有效的方法。

4. **寻找跨测试用例的重复工作。** 阅读测试运行的日志，看看子智能体是否都独立编写了相似的辅助脚本或采取了相同的多步骤方法。如果 3 个用例都导致子智能体编写了 `create_docx.py` 或 `build_chart.py`，这传达了一个强烈的信号：技能应该捆绑这个脚本。编写一次，放在 `scripts/` 中，并告诉技能去使用它。这能节省未来每一次调用时对“重复造轮子”的消耗。

这项任务非常重要（我们旨在创造巨大的经济价值！），你的思考时间并不是瓶颈；慢慢来，真正深思熟虑。我建议先写一个修订草案，然后换个角度审视它并改进。竭尽所能进入用户的角色，理解他们的需求。

### 迭代循环

改进技能后：

1. 应用你的改进
2. 将所有测试用例重运行到新的 `iteration-<N+1>/` 目录中，包括基准运行。
3. 启动评估视图，并使用 `--previous-workspace` 指向上一次迭代。
4. 等待用户回顾并告诉你他们已完成。
5. 阅读新的反馈，再次改进，不断重复。

直到：
- 用户说他们很满意
- 反馈全是空的（一切看起来都很好）
- 你没能取得实质性的进展

## 高级：盲测 (Blind Comparison)

在某些情况下（例如用户问“新版本真的变好了吗？”），为了进行更严密的版本对比，可以使用盲测系统。参阅 `agents/comparator.md` 和 `agents/analyzer.md` 获取详情。基本理念是：将两个输出给一个独立的智能体而不告诉它哪个是哪个，让其判断质量，然后分析胜出的原因。

这是可选的，需要子智能体。人类反馈循环通常已经足够。

## 描述优化 (Description Optimization)

SKILL.md frontmatter 中的 description 字段是决定 Claude 是否调用技能的主要机制。在创建或改进技能后，提出优化描述以提高触发准确性的建议。

### 第 1 步：生成触发评估查询

创建 20 个评估查询 —— 混合了应触发和不应触发的情况。保存为 JSON：

```json
[
  {"query": "用户提示词", "should_trigger": true},
  {"query": "另一个提示词", "should_trigger": false}
]
```

查询必须是真实的，符合 Claude 用户实际会输入的内容。不是抽象的请求，而是具体、详细的请求。包含文件路径、由于用户工作或背景而产生的信息、列名和数值、公司名称、URL。一点背景故事。可能包含拼写错误或随意的语言。

**差的示例**：`"格式化此数据"`、`"从 PDF 提取文本"`、`"创建图表"`

**好的示例**：“好吧，我老板刚才发给我这个 xlsx 文件（在我的下载文件夹里，叫什么 'Q4 sales final FINAL v2.xlsx' 之类的），她想让我加一列按百分比显示利润率。我记得收入在 C 列，成本在 D 列。”

对于**应触发 (should-trigger)** 的查询 (8-10个)，考虑覆盖面。对同一意图使用不同的表述 —— 有正式的，有随意的。包含用户未明确点名技能或文件类型但明显需要它的情况。

对于**不应触发 (should-not-trigger)** 的查询 (8-10个)，最有价值的是那些“差一点就触发”的情况 —— 分享了关键词或概念但实际上需要其他东西的查询。考虑相邻领域、朴素关键词匹配会触发但实际上不应该触发的歧义表述。

关键：不要让“不应触发”的查询显然不相关。针对 PDF 技能用“写一个斐波那契函数”作为负面测试太简单了 —— 它测试不出任何东西。负面用例应具有迷惑性。

### 第 2 步：与用户评审

使用 HTML 模板向用户展示评估集进行评审：

1. 从 `assets/eval_review.html` 读取模板
2. 替换占位符：
   - `__EVAL_DATA_PLACEHOLDER__` → 评估项的 JSON 数组
   - `__SKILL_NAME_PLACEHOLDER__` → 技能名称
   - `__SKILL_DESCRIPTION_PLACEHOLDER__` → 当前技能描述
3. 写入临时文件并打开。
4. 用户可以编辑查询、切换“是否触发”、添加/删除条目，然后点击“导出评估集 (Export Eval Set)”。
5. 文件由于浏览器设置通常下载至下载文件夹，请读取最新的 `eval_set.json`。

这一步非常关键 —— 糟糕的评估查询会导致糟糕的描述。

### 第 3 步：运行优化循环

告诉用户：“这需要一些时间 —— 我将在后台运行优化循环并定期检查结果。”

将评估集保存到工作空间，然后后台运行：

```bash
python -m scripts.run_loop \
  --eval-set <评估集路径> \
  --skill-path <技能路径> \
  --model <本场对话的模型 ID> \
  --max-iterations 5 \
  --verbose
```

使用系统中显示的当前模型的 ID，以便触发测试与用户的实际体验相匹配。

运行时，定期查看输出以告知用户当前处于第几次迭代以及评分情况。

这会自动处理完整的优化循环。它将评估集拆分为 60% 训练和 40% 持有测试，评估当前描述（每次运行 3 遍以获得可靠触发率），然后调用 Claude 根据失败项提出改进。完成时，它会在浏览器打开 HTML 报告并返回 JSON。

### 技能触发原理

理解触发机制有助于设计更好的评估查询。技能及其 name + description 会出现在 Claude 可用技能列表中。关键在于：Claude 只在遇到自己无法轻易处理的任务时才寻求技能帮助 —— 像“读取此 PDF”这样简单的一步式查询可能不会触发技能，因为 Claude 可以直接用基础工具处理。复杂、多步骤或专业化的查询在描述匹配时会可靠触发技能。

这意味着你的查询应该具有实质性内容。

### 第 4 步：应用结果

从 JSON 输出中获取 `best_description` 并更新 SKILL.md。向用户展示前后对比并汇报分数。

### 打包与呈现 (Package and Present) —— 仅在 `present_files` 可用时

检查你是否有 `present_files` 工具。如果没有，跳过此步。如果有，打包技能并呈现给用户：

```bash
python -m scripts.package_skill <技能文件夹路径>
```

打包后，引导用户前往生成的 `.skill` 文件路径以便安装。

- 草拟或编辑技能
- 运行测试提示词
- 与用户一起进行评估：
  - 创建 benchmark.json 并运行 `eval-viewer/generate_review.py` 帮助用户审查
  - 运行定量评估
- 重复直到满意
- 打包最终技能并返回给用户。
