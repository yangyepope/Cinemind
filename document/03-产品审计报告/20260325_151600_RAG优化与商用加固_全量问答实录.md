## 一、RAG 链条底层解析

### Q1：`rewrite_chain` 的作用是什么？

**背景**：代码位于 `core/rag_chain.py` L87-L92。

```python
self.rewrite_prompt = ChatPromptTemplate.from_messages([
    ("system", "根据对话历史和当前输入，生成一个【独立且具备完整语义】的搜索查询。仅仅输出重写后的查询字符串，不要有任何多余解释。"),
    MessagesPlaceholder(variable_name="history"),
    ("human", "{input}")
])
self.rewrite_chain = self.rewrite_prompt | self.llm | StrOutputParser()
```

**解析**：这段代码的作用叫做 **"问题改写 (Query Rewriting)"**，是 RAG 系统的关键防御层。

**核心问题**：在多轮对话中，用户往往会说指代不明的省略句：
- 轮 1：「请介绍影心的防晒产品」→ 检索正常
- 轮 2：「**它**的价格是多少？」← ⚠️ "它"是什么？向量库不知道！

如果直接把「它的价格是多少？」送去向量检索，会完全匹配不到任何有用内容。

**工作流程**：
```
用户输入 "它的价格是多少？"
       ↓
rewrite_prompt（注入历史记录 + 当前输入）
       ↓
LLM 理解上下文，改写为完整独立查询：
       → "影心防晒产品的价格是多少？"
       ↓
这个改写后的 standalone_query 才去向量库检索
```

代码中的具体体现（L103-L106）：
```python
data_prepare_chain = RunnablePassthrough.assign(
    standalone_query = self.rewrite_chain,        # ① 先改写问题
    context = (lambda x: x.get("standalone_query", x["input"])) 
              | self.retriever | format_docs       # ② 用改写后的问题去检索
)
```

**结论**：`rewrite_chain` 是一个专门处理"上下文依赖问题"的前置翻译层，将含有指代词、省略语义的问题转化为自包含、语义完整的搜索 Query，大幅提升召回准确度。

### Q2：每次都让大模型处理两次，性能如何优化？

**核心问题**：每次用户发送消息，LLM 会被调用 **2 次**：
1. 第 1 次：`rewrite_chain`（改写问题）
2. 第 2 次：`prompt | llm`（正式回答）

**代价与收益分析**：

| 维度 | 说明 |
| :--- | :--- |
| 代价 | 延迟翻倍 + Token 消耗翻倍 |
| 收益 | 多轮对话中检索准确率大幅提升 |
| 适合场景 | 用户提问上下文依赖性强（如"它"、"这个"） |
| 不适合场景 | 每轮都是独立完整问题（代价大于收益） |

**优化方案（短路机制）**：新增 `_needs_rewrite()` 函数，通过关键词检测做短路判断：

```python
# 指代词词典：包含这些词说明问题依赖上下文，需要 LLM 重写
_REFERENCE_KEYWORDS = [
    "它", "他", "她", "它们", "他们", "她们",
    "这个", "那个", "这种", "那种", "这款", "那款",
    "上面", "上述", "前面", "刚才", "之前说的",
    "这", "那", "此", "该",
]

def _needs_rewrite(input_text: str) -> bool:
    """
    短路检测器：判断用户当前输入是否包含指代词或省略语义。
    - 有指代词 → 问题语义不完整 → 必须触发 LLM 重写
    - 无指代词 → 问题自包含    → 直接跳过重写，节省一次 LLM 调用
    """
    return any(keyword in input_text for keyword in _REFERENCE_KEYWORDS)
```

**智能路由函数**：
```python
def smart_rewrite(x: dict) -> str:
    """智能重写路由：按需触发 LLM 改写，避免不必要的双重调用。"""
    if _needs_rewrite(x["input"]):
        logger.debug(f"[QueryRewrite] 检测到指代词，触发 LLM 重写 | 原始: {x['input'][:30]}")
        return self.rewrite_chain.invoke(x)
    else:
        logger.debug(f"[QueryRewrite] 问题自包含，跳过重写 | 原始: {x['input'][:30]}")
        return x["input"]

data_prepare_chain = RunnablePassthrough.assign(
    standalone_query = RunnableLambda(smart_rewrite),
    context = (lambda x: x.get("standalone_query", x["input"])) | self.retriever | format_docs
)
```

**效果对比**：

| 场景 | 优化前 | 优化后 |
| :--- | :--- | :--- |
| 独立完整问题 | 永远 2 次调用 | **1 次调用**（节省 50%） |
| 含指代词问题 | 2 次调用 | 2 次调用（保持准确） |

## 二、商用可行性审计全记录

### Q4：整个 RAG 项目能直接商用吗？

**结论**：⚠️ **不建议直接商用**，发现 4 个高危问题和 5 个中危问题。

#### 🔴 高危问题清单

**问题 1：`settings.py` 变量覆盖 BUG（最高优先级）**

```python
# L53-56：从环境变量读取（✅ 正确）
EMBEDDING_MODEL_NAME: str = os.getenv("EMBEDDING_MODEL_NAME", "text-embedding-v2")
LLM_MODEL_NAME: str = os.getenv("LLM_MODEL_NAME", "qwen-turbo")

# L89-91：硬编码直接覆盖以上内容（❌ 错误！导致 .env 失效）
EMBEDDING_MODEL_NAME: str = "text-embedding-v2"
LLM_MODEL_NAME: str = "qwen-turbo"
```
**影响**：`.env` 中的配置完全无效，无法切换模型。

**问题 2：会话历史无限增长（内存泄漏）**

`_history_store` 是全局字典，永不释放。100+ 用户并发时内存持续增长直至 OOM。

**问题 3：分支 A 关键词旁路知识库**

「帮我了解防晒产品」命中 `帮我`，直接走人设分支，知识库完全未被检索。

**问题 4：MD5 文件写入非线程安全**

多用户同时上传时，`mark_cache()` 追加写入不加锁，文件可能损坏。

#### Fix 2：`history_manager.py` — OrderedDict LRU 缓存 + 过期清理

```python
from collections import OrderedDict
from datetime import datetime, timedelta
from config.settings import SESSION_HISTORY_DIR, SESSION_CACHE_LIMIT

# 无界 dict → OrderedDict（支持 LRU 逐出）
_history_store: OrderedDict[str, BaseChatMessageHistory] = OrderedDict()

def session_history_store(session_id: str) -> BaseChatMessageHistory:
    if session_id in _history_store:
        _history_store.move_to_end(session_id, last=False)  # 移到头部=最近使用
        return _history_store[session_id]
    
    file_path = SESSION_HISTORY_DIR / f"{session_id}.json"
    _history_store[session_id] = FileChatMessageHistory(str(file_path))
    _history_store.move_to_end(session_id, last=False)
    
    # LRU 逐出：超过上限时淘汰最久未使用的
    while len(_history_store) > SESSION_CACHE_LIMIT:
        _history_store.popitem(last=True)
    
    return _history_store[session_id]

def cleanup_expired_sessions(expire_days: int = 30) -> int:
    """过期磁盘文件清理器，建议定期调用。"""
    threshold = datetime.now() - timedelta(days=expire_days)
    deleted_count = 0
    for session_file in SESSION_HISTORY_DIR.glob("*.json"):
        if datetime.fromtimestamp(session_file.stat().st_mtime) < threshold:
            session_file.unlink()
            _history_store.pop(session_file.stem, None)
            deleted_count += 1
    return deleted_count
```

#### Fix 5：`app.py` — 鉴权 + 限速

```python
from datetime import datetime

# --- 安全层 1：密码鉴权 ---
def _check_auth() -> bool:
    required_password = st.secrets.get("APP_PASSWORD", None)
    if not required_password:
        return True  # 未配置则开发模式放行
    if st.session_state.get("authenticated", False):
        return True
    # 展示密码输入界面
    password_input = st.text_input("请输入访问密码：", type="password")
    if st.button("确认进入"):
        if password_input == required_password:
            st.session_state.authenticated = True
            st.rerun()
        else:
            st.error("❌ 密码错误")
    return False

if not _check_auth():
    st.stop()  # 鉴权未通过，停止渲染

# --- 安全层 2：滑动窗口限速 ---
_MAX_REQUESTS_PER_MINUTE = int(st.secrets.get("MAX_REQUESTS_PER_MINUTE", 20))

def _check_rate_limit() -> bool:
    now = datetime.now()
    if "rate_limit_requests" not in st.session_state:
        st.session_state.rate_limit_requests = []
    # 清理 1 分钟以前的记录（滑动窗口）
    st.session_state.rate_limit_requests = [
        t for t in st.session_state.rate_limit_requests
        if (now - t).total_seconds() < 60
    ]
    if len(st.session_state.rate_limit_requests) >= _MAX_REQUESTS_PER_MINUTE:
        return False
    st.session_state.rate_limit_requests.append(now)
    return True
```

**配置方式**（`.streamlit/secrets.toml`）：
```toml

MAX_REQUESTS_PER_MINUTE = 20
```
