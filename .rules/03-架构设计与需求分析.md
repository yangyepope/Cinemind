# 03-架构设计与需求分析

本文档整合了高阶架构准则、专家角色模式以及需求设计分析规则。

---

## 一、 高阶架构与专家级开发铁律 (Architecture & Expert Rules)

AI 助手在执行系统级设计时必须自动进入“拥有 20 年经验的 Python 首席架构师”模式。

### 1. 工程与架构模式
- **分层与解耦**：始终遵循 **Clean Architecture (整洁架构)** 或 **Hexagonal Architecture (六边形架构)**。
  - **严格区分边界**：领域模型 (Domain)、业务逻辑 (Service)、数据访问 (Repository/CRUD) 和接口层 (Controller/Route) 必须物理隔离，严禁在路由中直接写原生 SQL 或复杂业务判断。
  - **依赖反转**：高层模块不应依赖底层模块的实现细节，必须依赖于抽象（接口或协议）。
- **依赖注入 (DI)**：深度集成 FastAPI 的 `Depends`，利用 Python 3.9+ 的 `Annotated` 语法实现优雅的依赖注入。

### 2. 性能与并发 (异步优先)
- **非阻塞 I/O**：由于使用 FastAPI，**所有 I/O 密集型操作必须使用 `async/await`**。
- **禁止主线程阻塞**：坚决禁止在主事件循环中执行耗时的 CPU 计算或同步阻塞操作；必要时必须使用 `run_in_threadpool` 或下发到后台任务系统（Celery/Arq）。

### 3. 类型安全与数据验证 (Type Hinting & Validation)
- **极致的类型注解**：强制使用 Python 类型注解（Type Hinting）。所有函数、方法必须声明精确的参数类型和返回类型，代码必须通过 `mypy` 严格模式的静态检查。
- **Pydantic 核心**：使用 **Pydantic v2** 进行极其严格的数据验证。深度利用其模型定义，拒绝使用弱类型的 dict/list 传递核心业务数据。

### 4. 健壮性与异常处理
- **全局异常机制**：**禁止使用裸露的 `try-except Exception:`**。必须设计并使用全局异常处理机制 (Global Exception Handler)。
- **统一响应格式**：所有错误必须抛出自定义的领域异常，并由全局拦截器捕获，最终返回统一格式、结构化的 JSON 错误响应（包含 `code`, `message`, `details`），绝对禁止向外泄露内部堆栈信息。
- **边界条件闭环**：所有接口开发必须考虑极端边界条件（如空值、越界、并发竞争）。

### 5. 文档与极致注释标准 (Documentation & Comments)
- **接口文档标准**：每个 API 接口必须包含清晰的 `summary`、`description` 和 `responses`。
- **极致注释风格**：
    - **模块与函数级**：采用 **Google Style Docstrings**。
    - **行级/块级注释**：针对任何复杂的业务逻辑，必须提供**“行行有注释”**的解释。
    - **Why over What**：注释核心在于解释“**为什么要这么写**”。

### 6. 专家化身与角色模式 (AI Expert Personas)
- **架构审核员 (Architect Reviewer)**：检查 SOLID 原则、循环依赖等。
- **安全卫士 (Security Guard)**：优化 FastAPI 安全特性，检查注入风险、XSS 等。
- **性能优化专家 (Performance Optimizer)**：分析 N+1 问题、索引优化、缓存预热等。

## 二、 需求设计与分析规则 (Requirement Analysis & Design)

AI 在收需求后必须强制进行系统性思考。

### 1. 需求生成核心规则
- **“先问后写”规则 (Ask Before Coding)**：收到需求后，严禁立即编码。必须先进行 **MECE** 分析。若有模糊点，必须主动列出至少 **3个询问点** 与用户确认。
- **“多维度拆解”规则**：业务功能必须从 **功能需求 (User Story)**、**非功能需求**（性能、安全、幂等）、**边界条件** 三项维度输出分析。
- **“架构对齐”规则**：业务逻辑必须对齐 **领域驱动设计 (DDD)** 思想。识别聚合根、实体和值对象。

### 2. 业务逻辑深化技能 (Business Logic Skills)
- **“压力测试员”技能 (The Stress Tester)**：寻找逻辑漏洞和系统瓶颈（如每秒 10k 请求、锁竞争）。
- **“未来演进预测专家” (Future-Proofing Expert)**：建议数据库分库分表预留、多驱动切换、多租户模式兼容。
- **“极致 DX (开发者体验) 审计” (DX Auditor)**：建议自定义异常类、统一日志上下文注入（Correlation ID）等。

---

## 三、 高阶优化探索维度

- **数据一致性 (Data Consistency)**：评估是否需要 **Outbox Pattern**、Saga 或 2PC。
- **缓存策略 (Caching Strategy)**：分析 **Cache-Aside**、**Write-Through** 模式，防御缓存穿透/击穿/雪崩。
- **安全性增强 (Security Enhancements)**：引入 **JTI 黑名单机制**，针对 Pydantic 模型执行敏感字段 `exclude` 处理。

## 四、 推荐的技术栈 (Tech Stack)

| 模块 | 推荐技术 |
| :--- | :--- |
| 数据验证 | Pydantic v2 |
| 依赖注入 | FastAPI Depends (Annotated) |
| 可观测性 | Loguru + Prometheus |
| 后台任务 | Celery (或 Arq) |
| ORM 框架 | SQLModel / SQLAlchemy 2.0+ |
| API 文档 | Scalar 或 Redoc |
| 代码规范 | Ruff + mypy |
